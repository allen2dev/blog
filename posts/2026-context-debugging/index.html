上下文、采样与线上排障：LLM 应用里没有「稳定复现」时你怎么查

2026年3月2日AI 与实践
AINode.js工程化

传统后端 bug 常常伴随堆栈与确定性输入输出；集成 LLM 的功能却可能出现：同一请求在两分钟内给出语义相近但不完全一致的回答。这不是玄学——通常是采样、上下文窗口拼接差异或上游权重漂移的综合结果。

这篇写给需要在 on-call 手册里写清楚「怎么查」的人：当稳定性不再是默认假设，观测体系要如何改写。

一、先分清三类失败 #

  1. 编排失败：超时、429、JSON 解析失败、工具调用异常——通常应在网关层分类打点；
  2. 模型质量问题：答非所问、幻觉引用——需要样本回放与 prompt 迭代；
  3. 数据质量问题：检索召回噪声——often RAG pipeline 的前半截出了问题。

若把一切笼统记为「AI 傻了」，你永远得不到有效复盘。

二、日志：trace id、prompt 版本与采样参数 #

最小日志集合通常包括：

  • trace id：贯穿网关、检索、模型调用；
  • prompt template 版本号（git sha 或 semver）；
  • temperature / top_p / max_tokens；
  • 检索命中片段 id（若是 RAG）。

没有 prompt 版本，你无法回答「昨天为什么对了」——只能在群里讲故事。

三、脱敏：聊天框也是数据入口 #

用户可能在对话里粘贴密钥、token、内部路径。日志脱敏策略应与保存银行卡号同级严肃——差别只是入口 UI 变成了聊天框。

四、前端：把诊断包装成一等交互 #

卡住的用户需要的不是一句抱歉，而是可复制给支持的最小复现包（脱敏后）：trace id、错误码、发生时间、客户端版本。

前端若能提供「复制诊断信息」按钮，支持团队的 MTTR 会明显下降——这不是模型能力问题，是软件工程体贴。

五、回归与回放：对抗漂移的长期战 #

权重漂移、prompt 微调、上游检索索引更新——任何一个变量都能改变输出分布。固定 golden set + 定期回放是唯一能对抗 drift 的工程手段；体感不能替代曲线。

结语 #

上下文窗口再长，也替代不了一份干净的 incident 记录。LLM 应用仍然是软件：可观测性、分级告警、责任边界——一件不能少。所谓「智能」，不该成为省略这些的理由。
