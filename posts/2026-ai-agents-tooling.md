近两三年最明显的变化，不是某个新框架，而是 **「模型 + 工具调用」** 成了默认心智：查文档、跑命令、改文件，都可以被编排成一步。兴奋之余，事故也多在「边界模糊」处发生——模型并不知道你们的上线窗口，也不会替你在周五晚上冷静。

作为写过十年脚本的人，我看到的机会与风险各一半：

- **把确定性留给脚本**：格式检查、构建、静态部署，仍然应该是可重复、可审计的命令。
- **把探索性留给模型**：接口调研、日志归纳、生成初版 CRUD——但要限定在分支与预览环境。
- **用契约锁住边界**：OpenAPI、类型定义、CI 红灯——Agent 再聪明也不应绕过。

```bash
npm run build && npm run preview
```

## 「编排」意味着什么？

意味着你要格外清晰地区分：**哪些是副作用（git push、发版），哪些是纯函数（lint、test）**。让人焦虑的不是模型写得多快，而是它能在你没看清时一连触发三步副作用。

未来会更吵（更多 agent、更多 MCP server）。能分清何时自动化、何时人肉拍板的人，会更吃香——这和 2016 年说「先想清楚再写代码」其实是同一句，只是键盘边上多了一只替你打字的手。
