近两三年行业最热的关键词之一是「Agent」——模型不再只输出文本，而是调用搜索、执行命令、改写仓库文件。听起来像程序员的终极外挂；落地之后最常见的痛点却是：**谁在深夜替一键串联的三步副作用买单？**

这篇文章不写 hype，只写一条朴素结论：**编排能力越强，边界纪律越重要**。老脚本手写年代犯的错，只是在更大音量下重复。

---

## 一、编排意味着什么？

所谓编排，是把若干步骤连成管线：**读上下文 → 决策 → 调用工具 → 观察反馈**。问题在于：**哪些步骤是幂等的？哪些是不可逆的？** 若这三件事没在设计文档里写清，Agent 再聪明也只是更快的 shotgun。

---

## 二、确定性留给脚本，探索留给模型

我仍然坚持：

- **格式化、lint、构建、部署到 staging**——脚本化、CI 化；
- **接口调研、日志归纳、草稿生成**——可以给模型更大自由度，但要在分支与预览环境。

这一条分界线一旦被涂抹，你会得到「周末某人让 Agent 改了 main」的传奇事故——技术上允许，组织上不原谅。

---

## 三、契约锁住边界：OpenAPI、类型、feature flag

模型不懂得你们的合规边界；**契约层（类型、OpenAPI、权限中间件）才是真正的守门员**。Agent 生成的代码若绕过契约，CI 应该红——这不是刁难，而是救生。

```bash
npm run build && npm run preview
```

这类命令看似简单，却是「最小可用验收」：**起码静态站点还能生成**，不至于演示赢了、合并炸了。

---

## 四、MCP 与插件宇宙：兴奋与维护成本

Model Context Protocol 一类扩展降低了集成门槛，却也引入新问题：**权限模型碎片化**。每个连接器都是一个潜在的 lateral movement 点——安全评审不能只盯着模型本体。

---

## 五、人的角色： orchestrator 仍是工程师

未来会更吵：更多 agent、更多 MCP server。稀缺的不是会按 Tab 的人，而是能分清「何时自动化、何时人肉拍板」的人——这和 2016 年说「先想清楚再写代码」并无区别，只是键盘旁多了一只替你打字的手。

---

## 结语

Agent 不会免除软件工程的基本定律：**可观测性、回滚策略、版本管理**。把它们写进流程里，模型才能更好的 acceleration——而不是 acceleration towards outage。
