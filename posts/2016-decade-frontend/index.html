从 jQuery 到组件化：一条战线上的四个阶段

2016年9月12日从业手记
前端随笔

如果把「前端」看成一条持续迁移的战线，我入行那年听到的最多的声音，大概是「兼容 IE」。那时候页面逻辑常常是：选中 DOM、绑定事件、在回调里改另一块 DOM。jQuery 解决的是跨浏览器 DOM 操作；它没有解决、甚至有意回避的问题是：状态散落在谁的脑子里。

这篇文章不想罗列框架编年史——那种表格网上已经够多了。我更想记录一种个人视角：同一个本质问题——「界面如何与数据保持一致」——在我们这一代人手里，如何被一次次重新表述。看清这条线，比记住某一年的流行名词有用得多。

第一阶段：回调与插件——「页面是一坨过程」 #

早期的典型写法，是在 $(document).ready 里注册一长串事件，再在回调里手动同步多处 UI。那时的复杂度主要来自两件事：DOM 查询成本高（所以要缓存 jQuery 对象），以及隐式依赖（谁先绑定谁后绑定会影响行为）。

插件生态一度极其繁荣：轮播、表单校验、无限滚动，全能「开箱即用」。问题在于：插件之间的状态从不往来。你在 A 插件里改了某个字段，B 插件里的计数不会自动更新——除非你再写一段胶水代码把它们缝在一起。胶水堆多了，就成了俗称的「意大利面」。

回头看，这一阶段教会我第一件事：没有单一数据源时，复杂度会指数级膨胀。不是 jQuery 错了，而是问题规模超过了「过程式补丁」能扛住的上限。

第二阶段：模块化——「文件边界开始出现」 #

CommonJS、AMD、后来 webpack 一统江湖，本质上在做同一件事：把代码切成文件，让依赖关系显式化。这一步的意义常被低估——很多人只记得「终于能写 import 了」，但真正换来生产力的是：你可以在 CI 里静态分析依赖图，可以在重构时 grep 到每一处引用。

我还记得第一次把模板从字符串拼写改成「模板函数 + 数据对象」时的解脱感。那不是框架的胜利，而是边界清晰的胜利：输入输出写在函数签名附近，评审同事一眼能看懂。

这一阶段教会我第二件事：模块边界比语法糖重要。边界对了，换 loader、换打包器都有路可退；边界糊了，换什么工具都像在废墟上刷漆。

第三阶段：MV* 与数据流——「谁拥有真相」 #

Backbone、AngularJS、后来的 Redux/MobX、Vuex/Pinia……名词层出不穷，争论也很多。但若剥掉名词，核心问题只有一个：谁是数据的唯一真相来源（single source of truth）？

一旦团队在这一点上达成共识，许多争论会自然消失：props 往下流、事件往上冒，或是单向数据流加 pure reducer——都是在不同约束下的工程答案。反过来，若有人在组件内部偷偷 fetch 又写全局变量缓存，另一人用事件总线广播副作用——你用的是最先进的框架，架构仍然是 jQuery 年代的隐性耦合。

这一阶段沉淀给我最重要的习惯：每次加功能前先问三个问题——数据存哪？谁有权写？读了会不会 stale？ 十年后的组件库换了几茬，这三个问题还在。

第四阶段：组件——「封装不变量」 #

组件化常被误解为「把 HTML 切成小块」。我更愿意把它定义为：对外暴露窄契约（props/events/slots），对内随意重构。契约稳定，协作成本就低；契约频繁破坏，团队就会在无尽的重命名与联调里耗尽耐心。

设计与实现上，这意味着：

  1. props 尽量「蠢」：最好是纯数据，而不是「半个 service」。
  2. 副作用收口：请求、埋点、订阅，集中在明确的生命周期或 composable 里，而不是散落在十个 watch 里。
  3. 样式边界：scoped、CSS Modules、原子类都可以，关键是「默认不外溢」。

结语：框架会过时，约束思维不会 #

回头看「从 jQuery 到组件化」，并不是一条「淘汰落后」的直线，而是问题规模变大之后，人类被迫发明更清晰约束的过程。下一波若又是什么新范式，只要还会问「状态谁拥有、边界在哪」，前面那些年就没有白写。

这条路上最大的敌人从来不是某个具体框架，而是侥幸心理：「先糊上线再说」。糊住的往往不是技术债，而是协作信任的存量——而那才是真正难还的利息。
