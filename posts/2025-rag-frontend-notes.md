内部文档问答几乎是每个团队在「尝试 AI」时的第一站：wiki、工单、Markdown 仓库一口气塞进向量库，demo 里问答流利——三周后却有人在 Slack 问：**「为什么它总能引用到陈旧段落？」**

这篇文章从我们一次次推倒重来的经验出发，聊一条更接近工程的路线：**别急着上最贵的向量数据库——先把数据流、评测与降级想清楚**。

---

## 第一章：问题定义比模型尺寸更重要

RAG（检索增强生成）本质是两段流水线：**检索器**选出候选片段，**生成器**基于片段回答问题。常见失败并不来自「embedding 维度不够」，而是：

1. **切块（chunk）策略错误**：太大则噪声多，太小则上下文不全；
2. **元数据缺失**：不知道段落出自哪个版本、哪个产品线；
3. **评测缺席**：每次改 prompt / 换模型只能靠体感。

我会在立项文档里强制回答：**用户允许的幻觉边界是什么？能否接受「不知道」？引用必须精确到什么程度？** 这些问题决定整条链路的 KPI。

---

## 第二章：最小可行路径——SQLite 也能跑第一轮

工程团队常被推销「要上向量库」。向量索引总有一天需要，但第一天更应该有的是：**干净的文本块与可追溯出处**。SQLite + JSON 存储向量在一百万 token 以内很多时候够用——足够你把整条链路跑通并收集失败样本。

典型流水线：

```text
原始文档 → 解析（标题层级）→ chunk → embedding → 存储
用户提问 → embedding → top-k → （可选 rerank）→ prompt → LLM
```

真正花时间的是 **chunk 以前**：表格如何展开？代码块是否单独切块？是否保留路径与时间戳？

---

## 第三章：重排序（rerank）什么时候值得加

top-k 纯向量检索在语义相近段落多时容易「张冠李戴」。cross-encoder reranker 往往能改善相关性——但要额外推理成本。取舍要看：**错误引用在你们场景下的代价**。代价若是合规风险，rerank 往往不是奢侈品。

---

## 第四章：前端如何把引用写成「可信」

后端能给出一串 chunk id，但若 UI 只展示一段摘要，用户仍会怀疑「你是不是编的」。更好的交互通常是：

- **展开引用原文**，折叠也行；
- **标注时间与文档路径**；
- **低置信度时显式提示**，而不是伪装确定。

前端团队常被低估在 RAG 项目里的价值：**你们最懂如何把检索结果变成负责任的界面**。

---

## 第五章：没有 golden question 集的 RAG 只是体感玩具

哪怕只有二十条固定问题，重复跑也能拦住八成 regression。评测不必一上来就上全自动打分——**human review + 结构化记录**已经足够让很多团队在第一个月止损。

---

## 结语

RAG 不是模型参数的炫耀场，而是**数据工程 + 信息检索 + 交互设计**的混合物。把地基夯实之前，换再大的向量库也只是昂贵复读机。
