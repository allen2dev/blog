团队第一次做「内部文档问答」时，很容易被名词淹没：**向量库、嵌入、重排序、Agent**。我的建议是：**先用一页 README 把数据流画清楚**，再决定要不要引入新基础设施。

### 最小可行路径

1. **Markdown / 工单** 按标题与章节切块（chunk），带上路径、更新时间等元数据。脏数据喂进去，再贵的向量库也只是昂贵垃圾回收站。
2. 调一次 embedding API，把向量与文本 ID 存起来——小到一定程度 **SQLite + JSON** 就够，不必一步到位 PG + pgvector。
3. 检索 top-k，拼进 prompt，让模型**归纳并引用片段**。前端最有价值的贡献往往是：**把引用展示得像引用**：段落链接、折叠原文、标注置信度。

```text
用户问题 → 检索 chunk → 组装 prompt → LLM → 带来源摘要的回答
```

当你发现回答经常「张冠李戴」，优先怀疑：**切块太大或太小、元数据缺失、评测集缺席**——而不是急着换更大的模型。

---

## 评测先于扩容

没有 golden question 集的 RAG，只能靠体感调参，三周后谁也说不清改了什么。哪怕只有二十条固定问题，重复跑一遍也能拦住八成 regression。

前端同学的优势恰恰是：**最懂如何把抽象检索结果变成可信 UI**。这比在后端多堆一个数据库更有长期价值。
