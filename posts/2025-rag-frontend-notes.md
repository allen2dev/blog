团队做内部知识检索时，很容易被名词淹没：**向量库、嵌入、重排序、Agent**。我的建议是：**先用一页 README 把数据流画清楚**，再选组件。

最小路径通常是：

1. **Markdown / 工单** 按标题与章节切块（chunk），带上元数据（路径、时间）。
2. 调一次 embedding API，存 **结构化 JSON + 索引**（小到一定程度 SQLite 都行）。
3. 检索 top-k，拼进 prompt，让模型**只负责归纳与引用**，不要让它「编造接口」。

```text
用户问题 → 检索 chunk → 组装 prompt → LLM → 带引用片段的回答
```

当你发现检索质量不够，再考虑换 embedding 模型或加 reranker——**问题多半在数据清洗**，不在「模型更大」。

前端同学的优势是：**最懂页面与交互如何把「引用」展示给用户**——这比在后端多堆一个数据库更有价值。
