入行头几年，页面多是「选 DOM、绑事件、手动同步状态」。jQuery 解决跨浏览器 DOM，却没解决**状态散落**：改一处数据，要在三四个回调里找谁该 `trigger`。

真正让我开窍的不是某个框架的名字，而是一种心智：**把 UI 看成状态的投影**。一旦接受这一点，你就会本能地问：单一数据源在哪？谁有权写？读路径是否可推导？这些问题后来贯穿了 Backbone、AngularJS，再到 React/Vue——语法在变，问题没变。

CommonJS、AMD 之争如今像史前遗迹，但当时能在 Node 里跑构建脚本，已经是生产力飞跃：**同一套模块图**，既能跑在服务端打包，也能跑在浏览器里。后来 webpack 接管了一切。有人说它重、配置繁琐——可工程一旦变大，**可预测的依赖图与缓存友好的拆分**，比「写得爽」更能保命。

> 经验之谈：先让模块边界清晰，再谈框架选型；顺序反了会一直在还债。

```javascript
// 早年 CommonJS 一页里的缩影：模板与状态分离，接口小而稳定
var template = require('./template.html')
module.exports = function render(state) {
  return template(state)
}
```

回头看，那段经历教会我两件事：一是**少抱怨工具链复杂**——复杂度往往来自业务与协作规模，不是 webpack 本身；二是**警惕「万能插件」**，它们往往在边界模糊的地方悄悄耦合。

---

## 组件化到底换来了什么？

组件不是「把 HTML 切成小块」。它的价值在于：**封装不变量**——输入 props、输出事件，内部实现怎么换都行。团队一旦共享这套契约，代码评审就不再纠结「这段字符串该不该出现在这里」，而是「契约是否仍然诚实」。

这条路走了十年，我依然认为：**框架会过时，约束思维不会**。下一波若换成别的范式，只要还会问「状态谁拥有、副作用在哪一层收口」，你就没有白写这些年前端。
