2023 年前后，编辑器里的补全从「猜下一个 token」进化成「敢写一整段」。团队的兴奋周期很短——几周后问题就变成：**谁在为一键接受的代码负责？**

我不会假装冷静：Copilot 类产品确实能把样板代码从十分钟压缩到三十秒。但若因此跳过设计与评审，你会用更快的速度制造 technical debt——只是债务人署名仍然是你。

这篇文章是我们内部沉淀的一套「手艺」：**如何让助手写得快，而你仍然掌控架构与风险边界**。

---

## 一、定位：实习生，不是合伙人

我把 AI 补全类比成熟悉语法的实习生：**打字快、语法错误少，但对你们的领域约束、合规条款、昨晚线上的事故一无所知**。合伙人会为架构债务负责；实习生不会——Copilot 更不会。

因此默认立场是：**它可以起草，不可以裁决**。

---

## 二、提示工程在工程团队里长什么样？

不是堆砌形容词，而是提供**确定性的脚手架**：

1. **先写函数签名与注释意图**，再让它填空——空白越大，幻觉越大。
2. **给定输入输出示例**（尤其是边界条件），比笼统描述「帮我实现」可靠得多。
3. **限定上下文**：贴相关类型定义与错误信息，而不是把整个仓库粘贴进去。

下面这种模式在我的仓库里反复出现：

```typescript
/**
 * 将用户草稿转为发布 payload；不负责鉴权。
 */
export function toPublishPayload(raw: Record<string, unknown>) {
  return {
    title: String(raw.title ?? ''),
    tags: Array.isArray(raw.tags) ? raw.tags.map(String) : [],
  }
}
```

人类写清边界，模型填充细节——这比从零生成「一整篇文章式函数」稳定得多。

---

## 三、审查清单：哪些代码不允许一键合并？

团队内部可以列出红色清单，例如：

- **权限与认证路径**
- **金额、计费、退款**
- **PII 进出日志**
- **加密参数与密钥加载**

这些模块不是「写得慢」，而是**一旦错了代价极大**。在这里节省五分钟打字时间毫无意义。

---

## 四、工具链仍是守门员

生成代码仍然要走 `tsc`、`eslint`、单元测试——一道不减。有人戏称「AI 让 CR 变成形式主义」——恰恰相反，**CR 应该更严肃**：因为你面对的 diff 体积更大、看似合理实则偏离需求的概率更高。

我会额外建议在 PR 模板里加一句：**若包含生成代码，请注明提示摘要与人工核对点**。

---

## 五、责任归属：按下合并键的人仍然是你

法律与职业伦理层面，维护者对合并入主干的行为负责。模型不知道你们的合规条款，也不会读你们的 on-call 手册。

所以 Copilot 改变的是**草稿速度**，不是**问责链条**。这一点，十年前后都一样——变的只是草稿来得更快，评审压力更大。

---

## 结语

真正稀缺的不是打字速度，而是**问题是否被正确定义**。把手艺写在流程里：提示边界、审查清单、CI 守门——生成式工具才能成为加速器，而不是放大器。
