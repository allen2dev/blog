2023 年起，编辑器里的补全从「猜下一个 token」变成了「敢写一整段」。兴奋过后，团队里很快出现两类人：一类把 Tab 按成肌肉记忆，另一类坚持每一行都读完再接受——后者的 velocity 看起来更低，事故率却往往不是一个量级。

我的用法很朴素：**把 AI 当成熟悉语法的实习生**——写得快，但不替你背架构决策。

### 实践中几条硬规则

1. **先写意图注释与接口签名**，再让它填空。空白越大，幻觉越大；边界清晰的函数更容易被「填满」而不是被「发明」。
2. **敏感逻辑必须人工审**：权限、金额、隐私数据路径、加密参数——一律不让「一键接受」。这不是不信任模型，而是不信任上下文窗口外的业务约束。
3. **生成代码仍走 CI**：`tsc`、`eslint`、单元测试，一道不减。AI 降低的是打字成本，不是评审义务。

```typescript
/**
 * 将用户草稿转为发布 payload；不负责鉴权。
 */
export function toPublishPayload(raw: Record<string, unknown>) {
  return {
    title: String(raw.title ?? ''),
    tags: Array.isArray(raw.tags) ? raw.tags.map(String) : [],
  }
}
```

## 责任边界在哪里？

法律与道德上，**按下合并键的人仍然是你**。模型不知道你们的合规条款，也不知道昨晚线上刚ban了哪种调用模式。

所以我更愿意把 Copilot 类产品看成「高频草稿生成器」，而不是「结对合伙人」。合伙人会为你的架构债务负责；Copilot 不会。

AI 改变的是**打字速度**，不是**问题是否被正确定义**。这一点，十年前后都一样——变的只是草稿来得更快，评审压力更大。
