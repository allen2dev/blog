Monorepo 实务录：当仓库变大以后，真正的敌人是隐式耦合

2023年8月21日前端工程
工程化MonorepoTypeScript

「都放在一个仓库里」听起来省事：共享代码抬手就 import，版本天然对齐。现实往往是几个月后出现另一类台词：「谁又改了底层类型？」「为什么 CI 绿了本地却红？」「这条依赖链到底是谁允许的？」

Monorepo 解决了一部分协作摩擦（例如跨包原子提交），却把另一类风险放大：边界含糊时的耦合速度。这篇文章从我亲历的项目出发，聊如何把单仓当成产品运维，而不是垃圾填埋场。

一、为什么我们会走向 Monorepo #

典型触发条件是应用多于三个、共享 UI / SDK 多于五个——否则独立仓库 + semver 发版反而更清晰。单仓真正的卖点通常是：

  1. 原子变更：一次 PR 同时修改调用方与实现，CI 能告诉你是否在编译层面闭环。
  2. 共享工具链：eslint/tsconfig/prettier 版本不对齐的概率显著下降。
  3. 代码所有权可视化：CODEOWNERS + path filter，让评审路径明确。

若你只是想把十个互不关联的业务强行塞进一个文件夹，收获的往往是 CI 缓慢与权限混沌——那不是 Monorepo，是单体地狱的数字升级版。

二、包的边界：可见性与依赖方向 #

几条我始终强调的规矩：

1. 每个包暴露一个窄 index #

其余路径视为 package-private——至少在文档与 eslint 规则里如此声明。

2. 禁止循环依赖 #

一旦出现 A→B→A，最后的解法常常是动态 import hack——那是架构认输的信号。

3. 区分「共享内核」与「业务模块」 #

内核应极少感知具体业务名词；业务模块之间默认不许横向穿透。

这些听起来像教科书——因为在规模化协作里，教科书问题就是事故源头。

三、依赖管理与幽灵版本 #

pnpm/npm/yarn workspaces 都能 workspace；差异往往在幽灵依赖（phantom dependencies）与 hoisting 行为。工具链升级一次，若 nobody 读过 changelog，就可能看见「本地能用、CI 挂了」的经典灵异事件。

我的最小建议是：锁文件进仓库、CI 使用 frozen install、依赖升级走单独 PR。别让依赖漂移和业务特性在同一次合并里出现——调试会变成刑侦。

四、CI：分层缓存与受影响路径 #

Monorepo 若每次都全量构建全仓，迟早有人在 Slack 里抱怨「流水线比午饭长」。常见做法是：

  • 受影响检测：基于变更路径触发任务；
  • 远程缓存：构建产物与测试结果被缓存；
  • 分层测试：lint/typecheck/unit/integration 分层失败定位。

工具（Nx/Turborepo/Bazel 思想）解决性能问题；架构纪律解决正确性问题——两边缺一不可。

五、何时不该 Monorepo #

若团队没有专职或兼职的「repo gardener」，且业务之间毫无共享内核——独立仓库往往更省心。单仓不是道德高地，只是组织能力与代码拓扑匹配时的一种选项。

结语 #

Monorepo 把「跨仓库发版」的摩擦换成了「跨包契约」的摩擦。契约清晰，你能放大协作收益；契约含糊，你会得到一个大号单体——只不过 blame 更难看了而已。边界清楚之前，别急着合并仓库。
