Vue 与 React：范式之争背后的六个实际问题

2021年3月2日前端架构
VueReact

互联网上每隔几个月就会复活一次「Vue vs React」大战：语法糖、Hooks、双向绑定、性能基准……热闹过后，落地项目的工程师往往在半夜面对的是另一类问题：谁在偷偷改共享状态？为什么这个 effect 又跑了三次？我们为什么要为一个表单引入三套抽象？

这篇文章不写胜负盘，只写我在两类栈里都踩过后总结的六个实际问题。它们与模板还是 JSX 无关，却决定你能不能按期交付。

问题一：团队的 muscle memory 在哪？ #

培训一个新同学的成本常被低估。若团队多数人闭着眼睛能写 React Hooks，突然转向 Vue 组合式 API——短期一定有产出损耗；反之亦然。

我的实用结论：团队已有强项优先于个人偏好。重写技术栈的真实成本往往在六个月后才显现（插件生态、招聘、历史包袱）。

问题二：组件边界如何定义？ #

Vue SFC 天然鼓励「单文件内聚合视图与逻辑」，对产品线迭代友好；React 函数组件 + Hooks 则倾向于「一切皆函数组合」。两边都能写好，也都写烂——关键在：你是否强制组件只做一件事，并在评审里拒绝「巨型组件」。

问题三：副作用住在哪？ #

React Hooks 把副作用写进 useEffect，优点是集中可见；缺点是依赖数组与闭包陷阱足够写一整本书。Vue 的 watch / watchEffect 也有自己的心智负担——尤其在异步取消语义不清晰时。

我更关心的评审问题是：数据请求与缓存策略是否在单一模块内可追踪，而不是用了哪种关键字。

function useArticle(slug: string) {
  const [data, setData] = useState<Article | null>(null)
  useEffect(() => {
    let cancelled = false
    fetchArticle(slug).then((d) => {
      if (!cancelled) setData(d)
    })
    return () => {
      cancelled = true
    }
  }, [slug])
  return data
}

示例虽小，却引出共同的纪律：竞态取消、slug 变化时的清理——这比争论 JSX 还是模板更接近工程质量。

问题四：生态债：谁是真正的集成成本？ #

动画、表格、国际化、可视化——往往需要第三方库。两边生态都足够成熟到支撑中大型应用，差异在于：你们的遗留模块最容易对接哪一边。这不是审美问题，是迁移路径长度问题。

问题五：测试策略是否同质？ #

无论是 Testing Library 风格的 DOM 测试，还是组件级快照——关键是团队有没有统一的「关键交互清单」。栈换来换去，若测试策略缺席，质量并不会自动变好。

问题六：替换栈的真实触发条件是什么？ #

我会严肃考虑换栈的场景通常只有几类：

  • 重大交互范式升级（例如大规模并发编辑需求）；
  • 招聘市场与外包生态剧烈偏移；
  • 现有栈的技术债已经阻塞安全更新。

单纯「酷」从来不是理由。

结语：范式之争之外的共识 #

Vue 与 React 都是在解决 UI=f(state) 的不同工程路径。框架争论永无止境；交付线上稳定版本的人，往往更在意边界清晰与可回滚。选一个栈，然后对它诚实——比来回跳槽体面得多。
