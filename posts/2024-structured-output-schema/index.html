结构化输出与 JSON Schema：别让我们的接口「写作文」

2024年11月7日AI 与实践
AILLMTypeScript

演示里最精彩的往往是模型口若悬河；上线后最先爆炸的往往是：下游解析不了、前端渲染不了、审计对不上。自由文本适合讲故事，不适合作为系统边界——除非你甘愿为每一类下游手写正则解析器。

这篇文章讨论一种更工程化的形态：让模型输出结构化 JSON，用 Schema 描述字段含义与约束，并在前后端共享同一套校验逻辑。

一、为什么自由文本在集成里是负债 #

当你需要：

  • 表格列字段绑定；
  • 数据库 NOT NULL 约束；
  • 权限系统按字段级别审计；

你就必须面对一个事实：字符串不是数据结构。团队若在 demo 阶段忽略了这一点，后期「补结构化」的成本常常高于重写集成层。

二、Schema 作为跨职能契约 #

JSON Schema（或 zod、protobuf、OpenAPI 组件 schema——选型视栈而定）的价值在于：人类与机器都能读。前端可以把 schema 编译成 TS 类型；后端可以用同一描述做运行时校验；模型侧则可以把 schema 附在 prompt 里作为输出约束。

const ArticleDraftSchema = {
  type: 'object',
  properties: {
    title: { type: 'string', maxLength: 120 },
    summary: { type: 'string', maxLength: 400 },
    tags: { type: 'array', items: { type: 'string' }, maxItems: 8 },
  },
  required: ['title', 'summary'],
} as const

这类约束看似啰嗦，却在评审会上消灭大量「字段叫什么」的争论。

三、失败路径要和成功路径同等设计 #

真实系统里，解析失败的比例往往足以单列监控指标。你需要预先回答：

  1. 重试策略：模型偶发输出夹杂 markdown code fence，是否 strip 再 parse？
  2. 降级：解析失败时是否 fallback 到人工编辑模式？
  3. 遥测：记录失败样本（脱敏）以供 prompt 迭代。

忽视失败路径的团队，会在第一周由用户帮忙「免费 fuzz」。

四、前端责任：把校验结果映射成可用 UX #

结构化输出的好处之一，是错误可以落到字段级——这正是前端所长：哪个输入框标红、附什么文案、是否提供「一键修复建议」。这比整页展示「模型胡说八道了」要体面得多。

五、与「模型负责创意，系统负责裁决」对齐 #

归根结底：模型擅长生成候选，校验器与业务规则负责裁决。Schema 是裁决席的外延——它把一部分裁决自动化了。

这和十年前做好表单校验没有本质不同，只是候选来源从键盘换成了模型。

结语 #

若你的 AI 功能仍停留在「返回一大段字符串让前端自己猜」，暂停一下，花半天画 schema——你会感谢未来某个半夜不用起床修解析 bug 的自己。
