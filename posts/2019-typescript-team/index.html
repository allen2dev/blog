TypeScript 在团队里落地的五年：从「可选」到「默认」

2019年11月20日前端工程
TypeScript工程化

第一次在全仓启用 TypeScript 时，团队的争论 rarely 关于类型系统本身，而是：要不要停下业务两个月重写。事后证明这是伪命题——健康的采纳从来都是渐进的：新代码先 .ts，老代码边界补 .d.ts，痛点模块分批收紧。

这篇文章是我作为参与者而非布道者的复盘：TS 能带来什么、解决不了什么、团队要怎么组织才不流于形式。

第一部分：收益不等于「少打字」 #

外界常把 TS 说成「自动补全更强」。补全确实更好，但真正省钱的通常是这三件事：

1. 公共 API 的契约可视化 #

组件 props、SDK 方法、HTTP 客户端封装——这些地方一旦写错，代价在联调、集成测试甚至线上。类型签名即文档，而且随 CI 更新；Wiki 页面做不到这一点。

2. 重构时的辐射清单 #

重命名、改签名、discriminated union 收窄——tsc 能列出受影响调用点。比起全文搜索字符串，它显著降低「漏改一处」的概率。

3. 跨边界的一致性 #

前后端若共享 schema（OpenAPI / protobuf / zod），TS 能把「运行时校验」与「静态类型」连成闭环——这条路走到位时，扯皮会少很多。

第二部分：渐进式不是放任式 #

渐进式绝不等于「任何地方都可以 any 糊一下」。我们内部用过几条粗粒度规则：

  1. 对外 exported 的类型默认不允许隐式 any；对内模块可以阶段性放宽。
  2. 每个 sprint 清理若干条 @ts-expect-error——携带注释说明原因与到期条件。
  3. 第三方库类型缺失时，优先官方 @types，其次最小 .d.ts，最后才是本地 shim；并在 PR 里留下 issue 链接。

没有纪律的渐进式，终点通常是 AnyScript：编译过了，心智没过。

第三部分：interface 背后的协作含义 #

类型不只是编译器输入，也是团队沟通的词汇表。例如：

interface PublishOptions {
  draft?: boolean
  tags: readonly string[]
}

export async function publishArticle(
  slug: string,
  opts: PublishOptions,
): Promise<{ url: string }> {
  return { url: `/posts/${slug}` }
}

readonly 不只是编译提示，它在评审里传达：调用方不应就地 mutate 数组。这类细节堆积起来，就是「契约文化」。

反过来，若类型写得随意——到处都是交叉类型大杂烩、union 从未穷尽——那么 TS 只会复制混乱，而不是消灭混乱。

第四部分：TS 解决不了的三类问题 #

第一，错误的需求建模。类型再漂亮，业务规则错了，只是在错误的地基上盖类型安全的楼。

第二，性能与分布式一致性。TS 不会帮你选缓存策略，也不会告诉你分布式事务怎么做。

第三，组织问题。若没人 Code Review、没人维护共享类型、没人更新 breaking change 的迁移指南——类型只会变成另一层需要专人维护的资产。

第五部分：工具链与开发者体验 #

严格模式、strictNullChecks、路径别名、项目引用（project references）——每一项都值得单独一次 tech talk。我的经验是：先把 CI 里的 tsc 跑稳，再谈 fancy 插件。地基不稳时叠加快门工具，只会让失败更难排查。

结语 #

TS 不是银弹，但是极好的安全带——前提是团队承认：类型的成本主要在协作与维护，而不是键盘。把它当成静态版本的 API 设计来对待，你会少走很多弯路。
