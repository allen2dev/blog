LLM 评测台搭建手记：别让「我感觉好一些」当 KPI

2025年9月14日AI 与实践
AI前端生产力

模型提供商偷偷更新权重、团队微调 prompt、某人顺手改了 temperature——三件互不商量的事可以在同一天发生，然后你在复盘会上听到：「我记得上周还挺聪明的」。这不是玄学，是缺少基线。

这篇文章写给需要在迭代里 defend budget 的工程师：如何把评测从「玄学体感」推进到「可重复的实验室流程」——哪怕第一期简陋得像作坊。

一、为什么体感不够用 #

人类记忆对「变好」敏感，对「变差」迟钝——尤其是文案类输出。没有固定问题集，你无法回答：昨天的改动是否伤害了主干场景。体感在第一周有用，第三周开始变成噪声。

二、Golden questions：宁可少，但要分层 #

不需要一上来两百条。更有效的是 二十条覆盖主干 + 已知翻车点：

  • 主干：最常见用户任务；
  • 边界：长上下文、空输入、恶意注入；
  • 回归：历史上出过事故的 ticket。

每条记录期望行为：允许的自由度（例如摘要长度）、禁用行为（例如编造接口）。

三、脚本先于仪表盘 #

我见过最务实的起点：

  1. 一个 markdown 文件列出问题；
  2. 一个 npm script 调模型 API，把原始回答落盘；
  3. git diff 对比两次运行目录。

复杂可视化以后再加。先有基线，再谈自动化打分。一上来就上多维雷达图往往死在没人维护特征权重。

四、并排 diff 胜过平均分 #

平均分掩盖大量失败模式。并排展示两次回答能让评审会迅速对齐「哪种退步更不能接受」——有时语义准确率没变，但冗长度暴增，对 UX 同样是灾难。

前端在这里的角色非常关键：把抽象分数变成可排序、可过滤、可分享的表格与 diff 视图——这比后端多写一个 REST 更能加速团队决策。

五、把评测纳入 CI 的策略 #

全自动打分门槛高，但可以做 lighter 的几件事：

  • schema 校验：输出仍是合法 JSON 吗？
  • 长度与禁用词：有没有触发红线？
  • 快照对比：与 baseline 的差异是否超过阈值触发人工队列？

结语 #

评测不是为了炫技，是为了在模型 silently drift 时你还有刹车。体感可以是起点，不能是终点——除非你甘愿每次都靠值班工程师的记忆兜底。
