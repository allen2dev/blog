CSS 架构十年札记：当样式成为「看不见的技术债」

2017年5月30日前端工程
前端CSS工程化

很长一段时间里，前端面试喜欢问 JavaScript，却很少认真问 CSS——仿佛样式只是「锦上添花」。等到页面过了三十屏、主题换过三代、设计师离职两轮之后，团队才会在某个深夜共识：CSS 可能是仓库里最贵的隐形债务之一。

这篇文章记录我亲历过的几种「救命药方」，以及它们各自适用的边界。不讲对错，只讲在什么约束下用什么东西才不后悔。

全局样式的诅咒：特异性战争 #

早期最常见的问题是 #header .nav li a:hover 这类选择器层层加码——不是为了炫技，而是为了「打赢」前面某人写下的规则。特异性一旦变成军备竞赛，重构就变成考古：你得先读懂三年前哪位同事在和谁较劲。

那时候我们学到的第一课，其实不是「少用 !important」，而是：团队的 CSS 也需要架构。没有架构的样式层，最终会变成一团所有人都不敢动的毛线球。

BEM：用命名买可读性 #

BEM 的核心贡献不是「三个下划线」，而是强迫你在命名时回答「块—元素—修饰符」这三个角色。冗长吗？冗长。但在没有构建工具魔法的日子里，它至少让 grep 与 Code Review 仍然可行。

实践中的坑在于：团队是否共享同一套边界定义。有人把「块」理解成页面模块，有人理解成组件——定义不一致时，BEM 只会把分歧编码得更长。

我会建议在引入 BEM 的同时写一页极短的约定：哪些属于 Block、何时新开 Block、修饰符能否跨组合。字数不多，但能避免半年后全员重构一轮类名。

预处理器：嵌套是蜜糖也是毒药 #

Less/Sass 带来了变量、嵌套、mixin。变量解决「颜色散落八十处」是真的；嵌套解决「写起来爽」也是真的——但若缺少纪律，嵌套会把特异性问题藏得更深：你在第五层嵌套里改了一个颜色，三个月后发现某个兄弟选择器仍然覆盖了你，因为编译后的选择器比你想象的更长。

我的做法一贯粗暴：嵌套不超过三层；再往下必须新开块或拆 class。这不是语法限制，而是给未来维护者留一口气。

CSS Modules 与作用域：边界终于进了构建链 #

当构建工具能在编译期生成唯一类名，「样式默认局部化」才真正成立。从此评审的关注点可以从「会不会误伤全局」转向「这块组件是否暴露了合理接口」。

代价是：动态类名组合需要约定（例如 styles.active 与模板字符串），以及**主题与令牌（design token）**必须提前设计。否则你会看到一堆 styles.xxx 与零散的 token 函数混用——边界清晰了，心智模型却分裂了。

原子类 / Utility-first：快与乱的辩证 #

原子类框架解决的是「重复写小型样式规则」和「设计代币强制一致」。在迭代极快、页面高度同质的产品里，它常常显著缩短从设计到实现的距离。

但它不是免费午餐：设计体系不成熟时，原子类只是把「随意拼颜色」变成了「随意拼 class」；可读性也会下降——长串 class 对新人并不友好。团队若选这条路，必须配套：

  1. 一套稳定的 spacing / color / typography token；
  2. 组件库承载复杂交互，页面层尽量组合而非发明；
  3. Code Review 里有人专职盯「是否发明一次性抽象」。

我至今仍遵守的三条底线 #

第一，全局入口必须清单化：reset、字体、基础变量可以全局；其余默认组件私有。
第二，魔法数字进变量：间距与色值若散落，设计一改你就输了。
第三，跨团队变更走 RFC：主题升级不是「替换变量文件」一句话，而是接口变更——要有迁移窗口与回归清单。

结语 #

CSS 从来不是「会不会写选择器」那么简单。它是协作协议的可视化层：谁在什么粒度上拥有样式主权，如何在迭代中不踩别人地里——这些问题若不事前说好，后期只能靠加班与补丁维系。

十年过去，我依然相信：约束清晰比工具新潮重要。工具会变，约束思维不会。
