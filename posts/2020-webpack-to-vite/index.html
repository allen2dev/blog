Webpack 之后：我为什么仍感激它，又愿意拥抱 Vite

2020年7月14日前端工程
ViteWebpack工程化

Webpack 在我职业生涯里占据的时间远比我想承认的长。它教会我「一切皆模块」与插件哲学——这套心智迁移到任何打包器都有用。但当仓库膨胀、入口增多、本地冷启动逼近一分钟时，我开始诚实面对一个问题：开发体验算不算生产力？

这篇文章记录一次典型的工具换代决策：不是追逐网红配置，而是量化痛点、列出迁移清单、设定回滚条件。

第一章：Webpack 到底赢在什么 #

在浏览器原生 ESM 尚未普及的年代，webpack 承担了三件事：

  1. 把任意资源纳入模块图：图片、字体、样式，皆可 import。
  2. 代码拆分与按需加载：路由级 chunk、vendor split，都有成熟范式。
  3. 插件生态：从 babel 到 TS，从 CSS 到 Vue SFC，几乎总能找到「loader + plugin」组合拳。

这些能力让它在企业级项目里站稳脚跟。说它「重」的人抱怨的往往是配置复杂度，而不是「不该存在」——复杂度很多时候反映的是真实世界的多样性。

第二章：痛点从何而起 #

当痛点不再是「能不能打包」，而是「改一行要等多久反馈」，争论焦点就变了：

  • 冷启动：模块图越大，初始化越久。
  • HMR：改动传递到浏览器的路径越长，心智越容易断裂。
  • 缓存：增量编译一旦不稳，开发者会养成「动不动就重启 dev server」的恶习。

这时候替代方案的价值主张通常是：开发路径走原生 ESM + 按需编译，生产路径仍交给成熟打包器（如 Rollup）。Vite 走的就是这条路——我不是在给某个商标背书，而是在描述一类架构。

第三章：迁移策略——顺序错了会翻车 #

经验里的安全顺序是：

1. 环境与静态资源 #

对齐 import.meta.env、process.env shim、public 目录行为，尤其是部署到 子路径（GitHub Pages）时的 base。我见过不止一例迁移卡在「线上资源 404」——那不是框架 bug，是路径假设变了。

2. 样式与 CSS 方案 #

Less/Sass、CSS Modules、PostCSS——逐项对照迁移前后输出是否一致。样式是最容易被低估的土雷。

3. 入口与代码分割 #

动态 import()、魔法注释、legacy bundle——对照产物体积与运行时请求瀑布。

4. Node polyfill 与浏览器 API #

旧代码若隐含 Node API，在新工具里可能直接报错——这正是「迁移清单」存在的意义。

npm run dev
npm run build

这两条命令应当成为迁移验收的最低门槛；在此之上再加 Lighthouse、e2e。

第四章：换代的标准是什么？ #

我会用三个量化维度说服自己：

  1. 本地反馈时间：从保存文件到浏览器更新的 P95。
  2. CI 构建时间：是否下降或至少不明显变差。
  3. 维护成本：配置文件行数、插件数量、新人上手文档厚度。

若只有第一项改善而后两项恶化，就要警惕：是否在把复杂度转嫁给 CI 与 onboarding。

第五章：工具换代背后的组织问题 #

任何构建工具都与 CI、Docker、预览环境耦合。换代从来不只是改 package.json，而是：

  • 缓存目录是否纳入 .gitignore / CI cache；
  • Source map 上传策略是否变化；
  • 线上错误监控里的文件名映射是否仍然成立。

忽视这些，就会出现「我本地飞快、同事全红」的经典悲剧。

结语 #

Webpack 与 Vite 之争，本质是开发路径与生产路径是否要用同一套权衡。无论选哪边，我都会坚持：构建管线服务于交付节奏，而不是反过来。工具会变，但「量化痛点—列出清单—保留回滚」这套决策骨架不会过时。
